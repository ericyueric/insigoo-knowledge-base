# LLM Wiki 编写标准

> **版本**: v1.0
> **目标**: 让文档既人类可读，又AI可读

---

## 为什么需要"LLM友好"？

传统文档（Word/PDF）给人看没问题，但AI读不懂：
- 格式封闭，AI无法解析结构
- 图片/表格信息丢失
- 无法自动索引和检索

**LLM友好的文档** = 纯Markdown + 结构化 + 元信息完整

这样的文档：
- ✅ 人类直接阅读（任何编辑器都能打开）
- ✅ AI能解析结构和语义
- ✅ 可自动向量化、检索
- ✅ 可跨工具迁移（不绑定平台）

---

## 五大原则

### 原则1：纯Markdown格式

**只使用 `.md` 文件**，不用docx/pdf/txt。

| 格式 | 是否推荐 | 原因 |
|------|---------|------|
| `.md` (Markdown) | ✅ 推荐 | 结构化、纯文本、AI友好 |
| `.docx` (Word) | ❌ 不推荐 | 二进制格式、AI难解析 |
| `.pdf` | ❌ 不推荐 | 排版固定、难以提取结构 |
| `.txt` | ⚠️ 不推荐 | 无结构标记 |

**特殊情况**：如果有现成的docx/pdf，可以：
1. 转换为Markdown（用pandoc等工具）
2. 保留原件在 `resources/` 目录
3. 在 `.md` 中引用原件路径

### 原则2：结构化标题

用 `#` `##` `###` `####` 组织文档层级：

```markdown
# 文档主标题

## 一级章节

### 二级章节

#### 三级章节
```

**规则**：
- 一个文件只有一个 `#`（主标题）
- 层级不要超过4级（`####`）
- 标题要简洁、有信息量

**反例**：
```markdown
❌ 一大段文字没有标题
❌ 用加粗代替标题（**这是标题** ← 错误）
❌ 标题层级跳跃（# 直接跳到 ###）
```

### 原则3：元信息前置

每个文档头部必须包含元信息：

```markdown
# [文档标题]

> **创建**: YYYY-MM-DD | **更新**: YYYY-MM-DD
> **维护者**: [姓名/角色]
> **状态**: 草稿 / 正式 / 归档

---

[正文内容]
```

**为什么需要元信息？**
- 知道谁写的、找谁问
- 知道何时更新、是否过时
- 知道文档状态（草稿/正式/归档）

### 原则4：原子化

**一个文件一个主题**，不要把所有东西塞一个文件。

| ✅ 好的做法 | ❌ 坏的做法 |
|-----------|-----------|
| `会议纪要/2026-06-25_月度例会.md` | `所有会议纪要.md`（500行） |
| `projects/项目A/计划.md` | `项目A的全部文档.md` |
| `policies/财务制度.md` | `制度汇编.md` |

**判断标准**：
- 如果一个文件>500行，考虑拆分
- 如果一个文件涵盖多个不相关主题，拆分
- 如果需要分别检索文件的不同部分，拆分

### 原则5：可链接

文档之间用相对路径互相引用：

```markdown
详见 [志愿者管理制度](../org/policies/志愿者管理.md)。

参考 [上次活动总结](../cases/2026-06-20_义诊总结.md)。
```

**好处**：
- 建立知识网络，不是孤立文件
- AI检索时能发现关联内容
- 减少重复（引用而非复制）

## Markdown 格式规范

### 标题

```markdown
# 一级标题（文档标题，每文件唯一）

## 二级标题（主要章节）

### 三级标题（子章节）

#### 四级标题（最小层级）
```

### 列表

```markdown
- 无序列表项1
- 无序列表项2
  - 嵌套项

1. 有序列表项1
2. 有序列表项2
```

### 表格

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 数据 | 数据 | 数据 |
```

### 代码块

````markdown
```python
# 代码块要标注语言
def hello():
    print("Hello")
```
````

### 引用

```markdown
> 这是引用内容
> 适合放重要提示
```

### 链接

```markdown
[链接文字](相对路径或URL)

[内部文档](../projects/项目A/README.md)

[外部网站](https://example.com)
```

### 强调

```markdown
**加粗**（重要内容）
*斜体*（术语解释）
~~删除线~~（过时内容）
```

## 好文档 vs 坏文档

### ✅ 好文档示例

```markdown
# 社区零废弃活动方案

> **创建**: 2026-06-25 | **更新**: 2026-06-25
> **维护者**: 张三（项目协调员）
> **状态**: 正式

---

## 活动概述

2026年7月15日在XX社区开展零废弃宣传活动，目标参与100人。

## 活动目标

1. 提升居民零废弃意识
2. 现场回收可回收物500kg
3. 招募长期志愿者20人

## 活动流程

| 时间 | 环节 | 负责人 |
|------|------|--------|
| 9:00-9:30 | 签到+开场 | 李四 |
| 9:30-10:30 | 知识分享 | 王五 |
| 10:30-12:00 | 现场回收 | 全员 |

## 注意事项

- 提前一周联系社区物业
- 准备回收袋、秤、记录表
- 参考上次活动总结：[2026-06-20活动总结](../cases/2026-06-20_活动总结.md)
```

### ❌ 坏文档示例

```markdown
零废弃活动

7月15日在社区搞活动，目标100人参加。要提升意识，回收500kg，招20个志愿者。9点开始签到，9点半分享，10点半开始回收。要提前联系物业，准备袋子秤和记录表。上次活动参考之前那个文件。
```

**坏在哪里？**
- 无标题结构（AI无法解析）
- 无元信息（不知道谁写的、何时）
- 无表格/列表（信息密度低）
- 链接不明确（"之前那个文件"是哪个？）

## 文档质量检查清单

写完文档后，检查：

```
□ 文件是 .md 格式？
□ 头部有元信息（创建日期/维护者/状态）？
□ 只有一个 # 主标题？
□ 标题层级合理（不跳级）？
□ 一个文件一个主题（不超长）？
□ 有链接到相关文档？
□ 表格/列表格式正确？
□ 命名符合规范（YYYY-MM-DD_主题.md）？
```

## 进阶技巧

### 1. 用注释隐藏备注

```markdown
<!-- 这是给维护者的备注，AI和读者通常会忽略 -->

# 文档标题
```

### 2. 用Front Matter（高级元信息）

如果需要更结构化的元信息：

```markdown
---
title: 社区零废弃活动方案
date: 2026-06-25
author: 张三
tags: [活动, 零废弃, 社区]
status: 正式
---

# 社区零废弃活动方案
```

### 3. 用锚点链接

```markdown
## 目录

- [活动概述](#活动概述)
- [活动目标](#活动目标)

## 活动概述

内容...
```

### 4. 用脚注

```markdown
零废弃是指尽可能减少废弃物产生[^1]。

[^1]: 定义来源：零废弃国际联盟
```
